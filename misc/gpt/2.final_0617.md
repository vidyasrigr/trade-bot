Critical gaps I would fix before paper trading
1. Add backend/data/tradier.py shim immediately
You clarified MarketData is the real source and tradier is just old naming. That’s fine. But multiple files still do:

from data.tradier import get_tradier
while the actual implementation is in:

data/marketdata.py
Affected paths include:

analysis/engine.py
analysis/whale_flow.py
analysis/liquidity_gate.py
analysis/cross_section_job.py
agents/graph.py
agents/position_monitor.py
agents/catalyst.py
scoring/portfolio_greeks.py
api/optimizer.py
api/freshness_check.py
Fix:

Create backend/data/tradier.py:

from data.marketdata import get_tradier, MarketDataClient
Do this before anything else. It is small, but it removes a whole class of runtime crashes.

2. Wire ticket_guards.py into the actual ticket path
scoring/ticket_guards.py is good. It checks:

earnings inside DTE
recent duplicate
cross-stream conflict
existing open position
But I do not see it actually enforced in the order-ticket path.

This is serious.

The code comment says it is used by agents/graph.py::_build_order_ticket, but _build_order_ticket() does not call run_all_guards().

Before paper trading, every emitted ticket should do:

flags = await run_all_guards(symbol, direction, stream, dte)
should_block, criticals = block_on_critical(flags)
And then:

critical flags block the ticket
warnings are shown on the ticket
all flags are logged
Without this, the system can recommend a trade right into earnings or stack duplicate same-symbol positions.

3. Wire recommendation_log.py into all outputs
You built a solid recommendation logging system, but I do not see log_recommendation() called from the main recommendation paths.

This is a huge gap because without it, the app cannot learn from itself.

Every generated recommendation should be logged before paper trading, even if the user does not enter it.

Why?

Because you need to track:

recommended but not taken
paper-filled
stale
blocked by guard
expired
won
lost
missed winner
avoided loser
The most valuable learning comes from all recommendations, not just trades manually opened.

Fix this before paper trading.

4. Your paper-trade open endpoint bypasses most system intelligence
/trades/paper/open checks:

circuit breaker
freshness
But it does not re-check:

ticket guards
liquidity
bid/ask spread
max contracts safe
earnings-in-window
duplicate exposure
recommendation id linkage
whether the ticket was actually generated by the system
whether the system still agrees with the trade
So paper trading can become manually-entered journaling, not system validation.

Before paper trading, I would require:

A paper trade must originate from a logged recommendation id.

Flow should be:

system emits recommendation
recommendation is logged
guards run
user clicks paper trade
paper trade links to recommendation
actual fill is recorded
monitor updates outcome
Right now that chain is incomplete.

5. PnL calculation for closing paper trades is too simplistic
Current close logic:

pnl = (exit_price - entry_price) * contracts * 100
That is fine for a simple long option, but wrong for many strategies:

short puts
short calls
credit spreads
iron condors
strangles
multi-leg trades
direction-specific exits
assignment-risk situations
For credit trades, PnL direction flips. For spreads, max loss/profit matters. For multi-leg structures, one exit_price is not enough.

Before paper trading options seriously, paper trades need a strategy_type and leg-level representation:

legs:
- C/P
- strike
- expiry
- qty
- entry bid/ask/mid/fill
- exit bid/ask/mid/fill
Otherwise paper results will be polluted.

6. The final app output is not backtested
This remains the biggest research gap.

You validate signals, yes. But production output is not:

signal -> trade
Production output is:

scanner
→ quick score
→ liquidity gate
→ catalyst flags
→ ranker tilt
→ LangGraph agents
→ trader thesis
→ adversary
→ risk manager
→ options selector
→ order ticket
→ paper trade
That entire chain is not replay-tested.

So before trusting it, build:

backtest/strategies/briefing_replay.py
It should replay historical days and ask:

What would the morning briefing have recommended that day?

Then measure forward returns.

This is more important than adding any new signal.

7. LLM conviction is still too powerful
In agents/graph.py, the trader LLM sets conviction. The quant layer can cap it only if confirmation fails.

But I would make this stricter:

Quant conviction should be the ceiling. LLM can reduce or explain. It should not increase.

Current risk:

quant says 62
LLM writes a beautiful thesis and says 87
app ranks it highly
user trusts it
That is dangerous.

Fix:

state["conviction_score"] = min(llm_conviction, scoring.conviction_score)
Then let the adversary/risk manager reduce it further.

LLM should be analyst, not allocator.

8. Half-Kelly is too aggressive for this stage
KELLY_FRACTION = 0.50

That is too high for unproven live/paper behavior.

Even with good backtests, Kelly sizing assumes your edge estimate is accurate. In trading, it usually isn’t.

Before paper trading, I’d set:

KELLY_FRACTION = 0.10 or 0.25
MAX_POSITION_SIZE_PCT = very small
For paper trading, use realistic size but conservative risk. For first live-small later, use tiny size.

Especially because the promoted strategy is short-vol with large drawdown.

9. VRP strategy should not paper trade as naked strangles first
The only promoted strategy is naked short strangle style VRP. It has edge, but it is exactly the kind of strategy that can look great until it gets hit hard.

Before paper trading naked short vol, I would require:

defined-risk version
hard event filters
portfolio vega cap
max loss cap
regime stress filter
earnings exclusion
index-level vol shock detector
gap-risk simulation
I would not let the first paper phase be naked strangles.

Paper trade safer versions first:

iron condor
defined-risk credit spread
smaller notional
no earnings
no meme/high-gap names
no biotech
no illiquid chains
10. Backtest engine has a major portfolio realism issue
backtest/engine.py realizes PnL only on exit dates and builds equity from realized PnL.

That misses mark-to-market drawdown while trades are open.

For options, this is a big deal.

A short strangle can be deeply underwater before eventually recovering. If equity only updates at exit, drawdown may look better or weirdly timed.

Before promotion, the options backtester should produce daily mark-to-market equity, not only realized exit-date equity.

Track:

open trade marks every day
unrealized PnL
portfolio Greeks
margin usage
worst intratrade drawdown
max adverse excursion
max favorable excursion
Without this, drawdown and risk are incomplete.

11. MarketData historical chain cache is good, but slippage needs per-symbol calibration
The options engine uses slippage based on half-spread. That is good.

But paper/live trust needs per-name execution profiles:

SPY/QQQ spreads are tight
NVDA/TSLA are usually okay
small caps, biotech, and event names can be awful
far OTM strikes are often fake-liquid
stale quotes can mislead fills
Before paper trading, add per-symbol/per-expiry-bucket cost stats from cached chains:

avg spread %
median spread %
90th percentile spread %
OI
volume
quote staleness if available
Then reject trades where spread is too expensive.

12. Scanner Stage 1 can bias the whole system
Stage 1 favors:

strong 20-day momentum
volume spikes
near 52-week high
theme names get boost
This is fine for finding action, but it means the whole system is biased toward hot/crowded names.

That can be profitable in trend regimes and bad in reversals.

Before paper trading, track the scanner itself:

what symbols got eliminated?
did eliminated symbols later perform better?
does Stage 1 kill mean-reversion winners?
does it overweight crowded tech?
Scanner recall matters. A great Stage 4 cannot save you if Stage 1 filters out the real opportunities.

13. Signal registry is good, but enforcement must be harder
I like that signal_registry.py has promotion statuses and tests checking sandbox/feature-only behavior.

But enforcement should be centralized.

No signal should affect scoring unless:

promotion_status == promoted
AND influences_conviction == true
AND validation exists
AND freshness is acceptable
Right now the architecture intends this, but I’d add a hard runtime guard inside compute_final_score() or the analyzers so unpromoted signals cannot sneak into conviction through category raw scores.

14. The LightGBM ranker should stay disabled until you have enough real labels
The ranker architecture is good, but early ML ranking is dangerous.

If the feature store is mostly recent scanner snapshots, the model can learn scanner bias instead of market edge.

Before trusting ranker tilt, require:

at least 1–2 years feature history
walk-forward ranker validation
IC by month
IC by regime
calibration by decile
no single feature dominating
no leakage from future returns/features
Until then, cap ranker tilt much lower or keep it observe-only.

15. Feature store is good, but current daily snapshots start only when the app runs
_persist_feature_snapshot() writes today’s scanner features. That is good going forward.

But for backtesting from 2021, you need historical feature reconstruction or cached daily snapshots.

Otherwise briefing replay only works from the day you started collecting features.

So for a proper replay, you need either:

historical feature backfill, or
replay computation using historical OHLCV/options chains as-of each date.
Your backfill_feature_store.py exists, but I would treat its output as critical infrastructure, not optional.

19. Paper trading needs to measure rejected trades too
This is very important.

Do not only track trades taken. Track:

recommended
rejected by guard
ignored by user
user manually overrode
stale before fill
no fill
filled worse than expected
exited by rule

Otherwise you cannot answer:

Is the system bad, or did the user not follow it?
Are guards helping or blocking winners?
Are stale tickets causing losses?
Is slippage killing edge?

20. The daily briefing needs a “why not trade” section
For every top setup, show:

With reasons.

Example:

This trains trust. A system that says “do not trade” well is more valuable than one that always finds something.

That is the right direction.

The next winning move is not “more AI.”
The next winning move is:

Make every recommendation logged, guarded, realistically fillable, portfolio-aware, and replay-testable.

Once that is done, paper trading becomes meaningful. Without that, paper trading can waste months because you will not know whether losses came from bad signals, bad fills, bad sizing, stale data, user overrides, or missing guards.
